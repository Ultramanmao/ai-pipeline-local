# 8 GB Pipeline Benchmark: Eleven Models, One GPU — The Definitive Comparison

> Date: 2026-07-19 · GPU: RTX 5060 Laptop 8GB (sm_120) · CUDA: cu132 · Framework: ComfyUI / DiffSynth / Wan2GP / OpenMontage · OS: Windows 11 native
> Release: https://github.com/Ultramanmao/ai-pipeline-local/releases/tag/v11.0

## TL;DR

Over six weeks I got **ten different AI pipelines** running on a single 8 GB RTX 5060 Laptop — text-to-image, image-to-video, talking-head, and an orchestration layer. Every one produces working output. One more (Wan 2.2-TI2V-5B) does not.

This release is the **definitive comparison**: one table, real VRAM peaks, real per-step timings, real prompt-language performance, and a decision tree that maps "what I want" to "which pipeline".

**The three numbers that matter:**

| Need | Best pick | Why |
|------|-----------|-----|
| Fastest image | SDXL | ~5–10 s / image, ~4–5 GB VRAM |
| Best Chinese | Qwen-Image GGUF | only model that renders Chinese text natively |
| Best photoreal | Flux.1-dev GGUF + realism LoRA | skin, depth of field, Sony prompt |

**The one thing that changed:** Krea 2 Turbo finally ran locally at 89.7 s / image on profile 3 — killing the "blocked" status it held since v7.0.

**The pipeline that still doesn't work:** Wan 2.2-TI2V-5B, stuck at ~200 s / step on sm_120. If you have a 5000-series card, expect the same.

## The New Entry: Krea 2 Turbo int8 (was ❌, now ✅)

Krea 2 entered the logs in v7.0 as "blocked" — the GGUF download was truncated by an HuggingFace Xet CDN bug (file returned `tensor_count=1846835937283`). An int8 safetensors fallback (13.49 GB) was downloaded, but first run on profile 4 took ~189 s / step and was misdiagnosed as the same sm_120 kernel trap that killed Wan 2.2.

It wasn't. The root cause was **mmgp's async circular shuttle**: profile 4 keeps preloading and swapping 414 MB weight chunks between CPU and GPU on every step, bandwidth-bound on 8 GB.

**The fix:** drop to `profile_no=3` + `asyncTransfers=False`. Profile 3 auto-downsizes to 80% VRAM (~5.2 GB) and runs clean.

**Real numbers (profile 3, 1024²):**

| Step | Time |
|------|------|
| First step (cold) | 14 s |
| Warm steps 2–7 | 8–9 s |
| Final step | 8 s |
| **Total** | **89.7 s** |

**Two Python patches were also required:**

1. `Qwen3VLConditioner.forward` — the text encoder's bf16 weights land on CPU by default while `input_ids` are on CUDA → device mismatch. Lazy `.to(device)` in `forward` with an `_qwen_on_device` cache fixes it.
2. `pipeline.__call__` — added a `loras_slists is None` guard so no-lora calls don't trigger a crash in `update_loras_slists`.

Krea 2's position in the table: **not the fastest** (SDXL, Z-Image, and Flux.2 Klein all beat it), **not the cheapest** on VRAM, but a fast alternative path that avoids the GGUF/Xet pipeline entirely. On an 8 GB card, it earns its slot as a backup when the GGUF tooling misbehaves.

## Pipeline-by-Pipeline: What to Actually Use

### Fastest images → SDXL

The old workhorse. 5–10 seconds per image, VRAM under 5 GB. Prompt in English only. Quality is "good, not wow" — fine for batching, iterating, trying compositions. The default pipeline if you just need an image *now*.

### Biggest resolution → Z-Image Turbo

1312² native output in 4 turbo-distilled steps, ~14 seconds. Covers the gap between "standard" (1024²) and "poster-ready" (1536+). VRAM hits ~7 GB so close everything else before running. The v3.0 release documents two segfaults on first run — both from missing fp8 weight registration, fixed in two lines.

### Most VRAM-efficient → Flux.2 Klein 4B

4 GB model, ~5.5 GB peak, 1024² in 8 steps. Leaves the most headroom for a co-running LLM. Ships with native Qwen3 Chinese text encoding — no extra nodes. The v6.0 release documents the VAE trap: must use `flux2-vae` (128 channels); the standard `ae.safetensors` (16 channels) produces a "128 channels" mismatch crash that looks like a model error but is a VAE channel error.

### Best photorealism → Flux.1-dev GGUF + realism LoRA

The ceiling for 8 GB photoreal. 100 seconds per image, Sony-camera prompt, `flux-realism LoRA` at strength 1.0. Real skin texture and depth of field. **Realism comes from the LoRA, not from steps.** v2.0 release: the realism LoRA "makes or breaks" the pipeline — without it, Flux.1-dev is competent but generic.

### Best Chinese → Qwen-Image GGUF

Alibaba's 17B distilled to Q3_K_S (8.95 GB), runs at ~5.4 GB VRAM thanks to GGUF offloading. The only pipeline that renders Chinese characters natively in images. 240 seconds per image, moderate clarity loss from Q3 quantization.

**Two hard traps (v5.0):**
- **Text encoder must be Qwen2.5-VL-7B (hidden=3584)**, not Qwen3-4B (hidden=2560). Using Qwen3 produces no crash — it just silently degrades quality. The 2560-dimension mismatch never throws because ComfyUI doesn't validate the encoder-to-model pairing.
- **The VAE must be `qwen_vae`**. The standard `ae.safetensors` is misidentified as a WanVAE by ComfyUI's loader and silently broken.

### One super-app → Wan2GP

A single web interface that loads every image model on this list. Runs Flux.1-dev, Z-Image, Flux.2 Klein, Qwen-Image, Krea 2 from one browser tab. Video generation only works at the tiny 848×480 resolution (long videos still OOM). v9.0 release: the "VRAM Digger" bug where Chrome keeps grabbing GPU memory, and the Python 3.11 requirement (won't run on 3.13).

### The video pipeline that works → LTX-2B I2V

33-frame animated PNG in ~3 minutes, 768×512. The STG pitfall is well documented now: use `LTXVApplySTG(block_indices=29)` — full-layer STG (0–47) shatters faces. Pair with `LTXVStatNormLatent(factor=1.0)` and `strength=0.30` for coherent faces. Multi-segment relay (v4.0) achieves SSIM 0.99+ across 3 segments with no face deformation.

### Talking head → SadTalker

512² talking-head video from image + audio, ~2 minutes. v8.0 release: four Python-3.13 compatibility patches (basicsr build, numpy 2.x deprecation, torchvision removal, torch float conversion). GFPGAN face enhancer must be made optional or the whole chain breaks on Python 3.13.

### The assembly layer → OpenMontage

Doesn't generate anything. Orchestrates everything. Remotion + subtitles + TTS render in ~90 seconds on the CPU. v10.0 release: the ComfyUI `/prompt` API format silently broke (now requires `{"prompt":{"nodes":{...}}}`), PIL can't write MP4 (use ffmpeg), and the bundled WAN 14B video model (~28 GB) won't fit on 8 GB — but the Remotion layer alone makes OpenMontage worth installing.

### The blocker → Wan 2.2-TI2V-5B

5B parameter image-to-video. On sm_120 (RTX 5060 Laptop) every sampling step takes ~200 seconds because the Triton kernel isn't optimized for the new CUDA arch. Five GitHub workarounds tested (xformers, sdp, --novram, fp8, smaller res) — none bring it to usable territory on this card. v11.0 verdict: skip Wan 2.2 on 5000-series. LTX-2B covers the I2V use case at a fraction of the time.

## The Decision Tree

```
Start → Chinese or English prompt?
  ├─ Chinese + need rendered Chinese text → Qwen-Image GGUF, done
  ├─ Chinese, no text needed → Flux.2 Klein (native TE), done
  └─ English →
        ├─ Need photoreal / portrait → Flux.1-dev GGUF + realism LoRA, done
        ├─ Need big resolution (1312+) → Z-Image Turbo, done
        ├─ Need maximum speed → SDXL, done
        └─ Need lowest VRAM (co-run LLM) → Flux.2 Klein, done
```

```
Video:
  ├─ Need a short clip from an image → LTX-2B I2V (33f, ~3 min), done
  ├─ Need a talking head → SadTalker (~2 min), done
  ├─ Need a full composed video → OpenMontage + Remotion + LTX-2B materials
  └─ Wan 2.2-TI2V-5B → skip on 5000-series cards
```

## 8 GB: The Rules That Repeated

These five mistakes account for ~80% of the failures across all eleven pipelines:

| Rule | What it catches | Seen in |
|------|-----------------|---------|
| **Names must match exactly** | ComfyUI loads by filename pattern. A single wrong suffix = silent fallback to wrong model | v7.0 (Krea 2: two Wan 2.1 UNET files) |
| **VAE channel count matters** | 128-channel and 16-channel VAEs are not interchangeable. The wrong one crashes or silently degrades | v6.0 (Flux.2 Klein), v5.0 (Qwen-Image) |
| **Text encoder must match the model's hidden dimension** | Qwen-Image + Qwen3-4B = no crash, just a silently worse image | v5.0 (Qwen-Image) |
| **`--lowvram` or GGUF is non-optional on 8 GB** | Full fp16 weights + activation memory > 8 GB = instant OOM | v1.0, v2.0, v5.0, v3.0 |
| **Python version must match the package** | SadTalker needs 3.11; ComfyUI needs 3.13. Never mix. | v8.0, v9.0 |

## Lessons

| Lesson | Detail |
|--------|--------|
| **Speed and quality are inversely correlated on 8 GB** | SDXL (5 s) vs Qwen-Image (240 s) — same card, 48× difference |
| **Chinese support is a pipeline choice, not a prompt trick** | No English-prompted model renders Chinese text. Pick Qwen or Flux.2 Klein from the start |
| **Profile selection is a performance parameter** | Krea 2 profile 4 (189 s) vs profile 3 (89.7 s) — same model, 2× difference from one setting |
| **A blocked pipeline is often one flag away** | Krea 2 was "blocked" for days over a profile number. Wan 2.2 stays blocked because the kernel isn't optimized |
| **8 GB can do image generation, not (yet) video generation** | LTX-2B is the only I2V that's actually fast. Wan 2.2 proves why "works on paper" isn't the same as "works on this GPU" |

---

## Where This Leaves 8 GB

By the end of v11.0, eight pipelines are genuinely useful on 8 GB, one (Wan2GP) is useful for images only, one (OpenMontage) is CPU-only infrastructure, and one (Wan 2.2) is blocked. The 8 GB ceiling is now fully mapped — not by model size, but by the actual interaction between the CUDA architecture (sm_120), the model's kernel maturity, and the offloading strategy.

**Bottom line: 8 GB is no longer the bottleneck for image generation. The bottleneck is kernel support for the new GPU architectures. Image pipelines are done. Video pipelines are not.**
