# Z-Image Turbo on 8 GB RTX 5060: From Double Segfault to a 14-Second 4-Step Image

> Date: 2026-07-15 · GPU: RTX 5060 Laptop 8GB · CUDA: cu132 · Framework: DiffSynth-Studio + ComfyUI 0.25.0 · OS: Windows 11 native

## TL;DR

Z-Image Turbo is one of the fastest open-source image generators in 2026 — 4 steps, ~14 seconds, 1312×1312 output on an 8 GB card. But the path to getting it running on DiffSynth involved two segfaults in the loader, one broken key namespace, and a 3.5-hour model download.

The headline number: **4 steps, 14 seconds, 6.99 GB VRAM**.

This article documents both the ComfyUI route (already running) and the DiffSynth route (the one that required patches), with the key difference between them.

---

## What Is Z-Image Turbo

Z-Image Turbo is Tongyi-MAI's accelerated text-to-image pipeline: a 6.15B DiT + 4.02B text encoder, purpose-built for high-step-reduction. The Turbo variant is optimized to produce usable images in 4–8 steps instead of the 20–50 that most diffusion models need.

Two endpoints you can run it through:

| Route | Model file | Format | VRAM | Status |
|-------|-----------|--------|------|--------|
| ComfyUI | `zImageFp8_turboRev11.safetensors` (7.2 GB) | FP8 single file, ComfyUI repacked | ~7.5 GB | ✅ Running |
| DiffSynth | Tongyi-MAI/Z-Image-Turbo (30.6 GB, 7 files) | BF16 multi-shard, official | ~7.0 GB | ✅ Running (with patches) |

Both produce the same quality. ComfyUI is easier to start; DiffSynth is faster per step and more controllable once the loader works.

## The ComfyUI Route: Works Out of the Box

ComfyUI gets a pre-repacked FP8 file: `zImageFp8_turboRev11.safetensors` (7.2 GB). Drop it in `models/unet/`, wire the nodes, generate. Done.

The catch is the repacked format — and that's where DiffSynth breaks.

## The DiffSynth Route: A Key Namespace Fight

DiffSynth's `ZImageDiT` model loader expects standard DiT key naming:

```
to_q.weight, to_k.weight, to_v.weight  (separate)
all_x_embedder.weight
```

ComfyUI's repacked FP8 file uses a **merged key** with quantization metadata:

```
qkv.weight                          (q/k/v merged)
qkv.weight.comfy_quant              (quantization metadata)
```

These two formats are structurally incompatible. DiffSynth can't read a ComfyUI-repacked file, and vice versa.

The fix is to download the official BF16 model directly from the source:

```bash
modelscope download --model Tongyi-MAI/Z-Image-Turbo \
  --local_dir E:/WSL/tools/DiffSynth-Studio-main/models/Z-Image-Turbo
```

That's 30.6 GB across 7 files. Download took 3 hours 36 minutes on the local network.

## Two Segfaults in the DiffSynth Loader

Even with the correct model, DiffSynth's default loader crashes on this hardware. Both crashes are in `diffsynth/core/loader/`.

### Segfault 1: DiskMap Multi-Shard Merge

**File:** `diffsynth/core/loader/model.py`

When the loader tries to open a multi-shard model through `DiskMap(path, device, dtype)`, it segfaults at shard merge time with:

```
RuntimeError: Attempted to access the data pointer on an invalid python storage
```

The DiskMap wrapper assumes a single-file storage layout. Three shards don't fit that assumption, and PyTorch's storage layer throws an invalid-pointer error.

**Patch** — bypass DiskMap entirely, use `load_state_dict` directly:

```python
# OLD (broken):
DiskMap(path, device, torch_dtype=dtype)
# → RuntimeError on 3-shard merge

# NEW (working):
load_state_dict(path, torch_dtype=dtype, device=device)
# → loads real dict, no DiskMap involved
```

### Segfault 2: In-Context dtype Conversion on ~10 GB Shards

**File:** `diffsynth/core/loader/file.py`

The default loader reads each tensor and converts it to the target dtype inside the `safe_open` context:

```python
# OLD (broken):
with safe_open(path) as f:
    for k in f.keys():
        sd[k] = f.get_tensor(k).to(dtype)  # segfaults on ~10 GB shard
```

On the 9.97 GB transformer shards, calling `.to(dtype)` inside the context triggers a segfault — PyTorch can't materialize the dtype-converted copy of a 10 GB tensor within the same storage scope.

**Patch** — read everything first, then convert dtype:

```python
# NEW (working):
with safe_open(path) as f:
    for k in f.keys():
        sd[k] = f.get_tensor(k)       # read as-is
# then:
for k in sd:
    sd[k] = sd[k].to(dtype)           # convert after read completes
```

Both patches are 5 lines each. Together they turn a segfaulting loader into a working one.

## The Working DiffSynth Setup

```python
MODEL = "E:/WSL/tools/DiffSynth-Studio-main/models/Z-Image-Turbo"

vram = {
    "offload_dtype": torch.bfloat16, "offload_device": "cpu",
    "onload_dtype": torch.bfloat16,  "onload_device": "cpu",
    "preparing_dtype": torch.bfloat16, "preparing_device": "cuda",
    "computation_dtype": torch.bfloat16, "computation_device": "cuda",
}

pipe = ZImagePipeline.from_pretrained(
    torch_dtype=torch.bfloat16, device="cuda",
    model_configs=[
        ModelConfig(path=glob.glob(f"{MODEL}/transformer/*.safetensors"), **vram),
        ModelConfig(path=glob.glob(f"{MODEL}/text_encoder/*.safetensors"), **vram),
        ModelConfig(path=f"{MODEL}/vae/diffusion_pytorch_model.safetensors", **vram),
    ],
    tokenizer_config=ModelConfig(path=f"{MODEL}/tokenizer/"),
    vram_limit=7.5,  # 8 GB card, leave 0.5 GB margin
)
```

CPU offload is mandatory on 8 GB — BF16 full model (6.15B DiT) alone is ~12 GB on GPU. The `vram_limit` + CPU offload keeps it at ~7 GB.

## 4 Steps vs 20 Steps: What You Actually Lose

Z-Image Turbo's whole pitch is speed over iteration count. In practice:

| Steps | Time | Quality | Use Case |
|-------|------|---------|----------|
| 4 | 14s | Good — clean composition, correct subject | Drafts, iterations, batch runs |
| 8 | 28s | Recommended — sharper detail, richer texture | Production output |
| 20+ | 70s+ | Diminishing returns — marginal detail gain | Not recommended |

The 4-step output is already publishable for most use cases. Going to 8 steps is the sensible default. Pushing past 12 gives you texture refinement but costs 5× the time.

## Why This Matters on 8 GB

Most 2026 image models demand 12 GB+ to run at full precision. Z-Image Turbo is the exception: it's genuinely fast *and* fits on 8 GB when you use CPU offload correctly.

## The Three Pitfalls That Cost the Most Time

1. **DiskMap segfault on multi-shard models** — assume DiskMap is broken for multi-file models and skip it. Direct `load_state_dict` is safer.
2. **dtype conversion inside `safe_open`** — always separate read from convert on large files (>5 GB). The context manager holds storage that segfaults on in-place dtype ops.
3. **ComfyUI repacked ≠ DiffSynth compatible** — the key namespace is different. Don't try to share a single file between the two frameworks.
