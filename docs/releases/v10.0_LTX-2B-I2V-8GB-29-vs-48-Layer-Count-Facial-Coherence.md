## What Is LTX-2B

LTX-2B (ltx-video-2b-v0.9.5) is a text-to-video diffusion transformer at 2B parameters. It is one of the smallest models that produces coherent motion at 768Ă—512, and on an 8 GB RTX 5060 it is the only video model that runs at all â€” Wan 2.2, CogVideoX, and Bernini 14B all exceed 8 GB VRAM or fail on the sm_120 CUDA kernel.

The model works in two stages:

1. **Latent encoding** â€” the source image is encoded through a tiled VAE into a latent volume
2. **Diffusion sampling** â€” the latent is denoised over 30 steps with temporal attention regularization (STG) to prevent frame drift

The output at 49 frames / 24 fps is roughly 2 seconds per segment. Longer videos require relaying â€” chaining the last frame of one segment as the input to the next.

## The Three Mistakes

### Mistake 1: The Wrong Layer Count

The first version of this pipeline used `LTXVApplySTG(block_indices="[0,47]")` â€” all 48 transformer layers of the model.

The result was immediate and consistent: every frame had a distorted, melting face. The model was not failing at sampling. It was failing at attention regularization.

**Root cause:** STG (Self-Temporal Guidance) is applied as a per-layer attention penalty. The [0â€“47] configuration is tuned for LTX-13B, a 6Ă— larger model with a deeper transformer body. On LTX-2B, applying the same penalty across all 48 layers of a much smaller model suppresses facial features that the model barely has bandwidth to represent.

**The fix:** Use a single STG layer â€” `block_indices=29` â€” which matches the layer count and attention depth that the 2B model actually uses.

| STG config | Within-segment SSIM | Face | Motion |
|------------|--------------------|------|--------|
| `[0â€“47]` (13B preset) | 0.94 | âťŚ Severe distortion | Visible |
| `[29]` (2B preset) | **0.99** | âś… Clean | Minimal |
| `[14,33]` | 0.82 | âťŚ Drift | Visible |
| `[0,47]` via sampling-level | 0.93â€“0.95 | âś… Clean | Visible |

The `[0â€“47]` preset does not produce worse output by chance. It produces worse output by design â€” it is the wrong tool for the model size.

### Mistake 2: The Wrong STG Node

`LTXVApplySTG` is a **model-level** STG node â€” it patches the model directly at load time. `STGGuiderAdvanced` is a **sampling-level** STG node â€” it applies the regularization during each denoising step.

Only `STGGuiderAdvanced` works with `LTXVLoopingSampler`, the sampler that produces clean continuity. Using `LTXVApplySTG` with `LTXVLoopingSampler` creates a double-STG application that compounds the facial damage.

The correct node chain:

```
STGGuiderAdvanced
  block_indices: [0,47]
  stg_scale_values: 2, 2, 2, 2, 2, 0
  sigmas: 1.0, 0.9933, 0.9850, 0.9767, 0.9008, 0.6180
  cfg_values: 8, 6, 6, 4, 3, 1
  cfg_star_rescale: True

LTXVLoopingSampler
  temporal_tile_size: 80
  temporal_overlap: 24
  cond_image_strength: 1.0
  adain_factor: 0.0

LTXVStatNormLatent: factor=1.0
LTXVTiledVAEDecode

strength: 0.30 (face preserved) / 0.45 (visible motion)
seed: 777777
resolution: 768Ă—512
frames per segment: 49 (hard limit)
```

Note that `block_indices=[0,47]` is acceptable here because it is applied at the **sampling level** via `STGGuiderAdvanced`, not at the model level. The same number, a different node, and the result is clean.

### Mistake 3: DisableNoise Produces a Black Frame

`LTXVLoopingSampler` requires a noise source. The old pipeline used `DisableNoise`, which sets the noise tensor to zeros.

On LTX-2B, `DisableNoise` does not produce a clean initial condition â€” it produces an all-black frame, and the sampler has nothing to denoise from.

**Fix:** Use `RandomNoise` with `seed=777777`. This is the seed used in the baseline run that produced the first successful 145-frame relay, and it remains the most stable choice for this hardware.

## The strength Parameter: Where Face and Motion Meet

`strength` controls how much the latent moves away from the source image frame during sampling.

| strength | Within-segment SSIM | Face | Motion |
|----------|-------------------|------|--------|
| 0.10 | 0.97 | âś… | Head still rotates 30â€“45Â° (architecture limit) |
| 0.30 | 0.95â€“0.99 | âś… | Minimal (minimum usable) |
| 0.45 | 0.93â€“0.96 | âś… | Visible, acceptable |
| 0.50+ | < 0.90 | âš ď¸Ź | Segment melts mid-generation |

**There is no single strength value that gives both strong motion and a clean face.** This is an architecture-level limitation of LTX-2B, not a tuning problem. For 8 GB use cases, the choice is between `strength=0.30` (clean face, minimal motion â€” suitable for talking-head lock) and `strength=0.45` (visible motion, clean face with the correct STG configuration).

Attempts to keep the head completely still (`strength=0.10`) fail â€” the head rotates 30â€“45 degrees regardless. This is a model architecture constraint, not a prompt problem. For head-locking or camera control, SadTalker or AnimateDiff are the correct tools, not LTX-2B.

## Lessons

| Lesson | Detail |
|--------|--------|
| **A layer count is a model-size contract** | STG `[0â€“47]` is correct for LTX-13B, wrong for LTX-2B. The same configuration destroys one model and cleans another. |
| **Two nodes with the same name are not interchangeable** | `LTXVApplySTG` (model-level) and `STGGuiderAdvanced` (sampling-level) look like synonyms. They are not. Only the sampling-level node works with LTXVLoopingSampler. |
| **`DisableNoise` is not noise-free** | On LTX-2B it produces a black frame. `RandomNoise` with a fixed seed is the actual baseline. |
| **Strength is a trade-off, not a dial** | There is no setting that gives both strong motion and a clean face. Pick one and own it. |
| **49 frames is the hard limit** | Beyond 49 frames per segment, the middle of the segment melts. Relay is mandatory for anything longer than 2 seconds. |
| **Segment 3 is where the relay breaks** | Segments A and B are stable. Segment C accumulates enough latent drift to hallucinate at the final frame. |
| **LTX-2B cannot lock a head** | Even at `strength=0.10`, the head rotates 30â€“45Â°. This is model architecture, not a prompt or parameter issue. |
| **Stitch SSIM of 0.87â€“0.89 is seamless** | The human eye does not perceive the frame boundary at these scores. |

## The Real Story

The LTX-2B pipeline was not broken when it first ran. It was broken by the wrong number.

The first successful run produced 145 frames across three segments with clean facial coherence â€” and it used `STGGuiderAdvanced` with `block_indices=[0,47]`. Then the pipeline was migrated to ComfyUI 0.24.0 with a different set of built-in nodes, and `LTXVApplySTG` (the model-level equivalent) was used instead.

The new version produced the same output the 13B preset produces on a 2B model: melted faces. The difference between the two runs was not the model, the seed, the resolution, or the prompt. It was one node choice and one layer count.

A 4-segment table in a debugging spreadsheet told the full story:

| STG node | Layers | SSIM | Face |
|----------|--------|------|------|
| `LTXVApplySTG` | `[0â€“47]` | 0.94 | âťŚ |
| `LTXVApplySTG` | `[29]` | 0.99 | âś… |
| `STGGuiderAdvanced` | `[0,47]` | 0.93â€“0.95 | âś… |
| `STGGuiderAdvanced` | `[29]` | 0.96 | âś… |

Four rows. One integer. The rest is pipeline.

The question is no longer whether LTX-2B runs on 8 GB. It does. The question is whether the output is good enough for the use case â€” and for talking-head video, micro-expression content, and short relay sequences, it is. For anything requiring head lock, camera movement, or strong motion, it is not â€” and that is a model architecture answer, not a configuration answer.
