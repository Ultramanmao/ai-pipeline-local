# Qwen-Image on 8 GB: The 3584 vs 2560 Trap and Why "Best Chinese Text" Comes at a Cost

> Date: 2026-07-15 · GPU: RTX 5060 Laptop 8GB · CUDA: cu132 · Framework: ComfyUI · OS: Windows 11 native

## TL;DR

Qwen-Image 2512 is widely cited as the best open-source model for Chinese text rendering — logos, posters, dialogue bubbles, the works. On paper, it's a 20B MMDiT with a Qwen 2.5-VL 7B text encoder and a 1328×1328 native resolution. On an 8 GB card, it's a 56 GB model that doesn't fit.

The GGUF distillation route brings it down to ~9 GB for the UNet, but the real story is a dimension mismatch that costs two days: the model expects a text encoder with hidden size **3584**, and the Qwen3 GGUF you grab off HuggingFace has hidden size **2560**. ComfyUI doesn't error out. It just produces blurry images.

The headline numbers for the 8 GB viable path: **Q3_K_S GGUF, 5.4 GB VRAM, 1024×1024, ~150 seconds at 15 steps, quality 6/10**. The full-precision model needs 12 GB+ and is not this article's story.

This article documents the text encoder dimension trap, the working GGUF node chain, and the honest quality gap between the 8 GB route and what the model was designed to do.

---

## What Is Qwen-Image

Qwen-Image 2512 (December 2025) is Alibaba's open-source text-to-image model. At 20B parameters, it sits in the same league as FLUX but with one superpower: Chinese text rendering. The text encoder is Qwen 2.5-VL 7B — an actual LLM vision model, not the CLIP + T5 concatenation that most diffusion models use.

| Property | Value |
|----------|-------|
| Architecture | MMDiT (Multimodal Diffusion Transformer) |
| Parameters | 20B |
| Text encoder | Qwen 2.5-VL 7B (hidden dim: **3584**) |
| Native resolution | 1328×1328 |
| Full model size | 56 GB (transformer) + 16 GB (text encoder) + 0.2 GB (VAE) |
| Inference type | bf16 (full) / GGUF quantized (8 GB) |
| License | Apache 2.0 |

The text encoder is the key differentiator. Because it's a real LLM, Qwen-Image understands complex prompts in Chinese natively and renders Chinese characters inside the generated image. No other open-source model in 2026 does this at anywhere near the same quality.

## The Two Routes

### Route A: Full-precision (needs 12 GB+ GPU)

The full model loads via `DiffusersLoader` in ComfyUI. The workflow is clean — model, VAE, and text encoder all come from the same `Qwen-Image-2512` directory. Optimal parameters are counterintuitive:

| Parameter | Value | Note |
|-----------|-------|------|
| steps | 20 | More than 20 gives no visible improvement |
| CFG | 5.0 | 7.0 over-saturates and looks harsher |
| scheduler | normal | karras makes it blurrier |
| sampler | euler | Default |
| resolution | 1024×1024 | Verified stable |

With these settings, quality is 9/10. The full model is the real product. But on an 8 GB card, the transformer alone is 56 GB, and `--lowvram` disk offload crashes with segfaults during inference. This route requires a 12 GB or larger GPU.

### Route B: GGUF distillation (8 GB viable)

QuantStack provides a distilled + quantized GGUF:

| File | Size | VRAM | Quality |
|------|------|------|---------|
| Qwen_Image_Distill-Q2_K.gguf | 7.06 GB | ~5.5 GB | 5/10 |
| **Qwen_Image_Distill-Q3_K_S.gguf** | **8.95 GB** | **~5.4 GB** | **6-7/10** |
| Qwen_Image_Distill-Q4_K_M.gguf | 13.07 GB | ~7-8 GB | 8/10, but 8 GB will OOM |

The Q3_K_S variant is the practical sweet spot: it fits, it runs, and the remaining 100 MB of headroom on an 8 GB card is enough to keep PyTorch from crashing.

## The 3584 vs 2560 Trap

This is the part that costs time. The full Qwen-Image model uses Qwen 2.5-VL 7B as its text encoder, which has a hidden dimension of **3584**. The GGUF loader in ComfyUI-GGUF needs a matching text encoder file, and the obvious choice on HuggingFace is `Qwen3-4B-Q4_K_M.gguf` — which has a hidden dimension of **2560**.

Comedy: ComfyUI loads both files without complaint. `CLIPTextEncode` runs. The sampler runs. An image is produced. And it looks blurry.

The error only surfaces as a silent degradation. If you want to see the failure explicitly:

```
RuntimeError: normalized_shape=[3584], expected input with shape [*3584],
but got input of size [1, 31, 2560]
```

This error appears when you try to use `CLIPLoaderGGUF(type=qwen_image)` directly, because the type check catches the mismatch. Without the type flag, the wrong encoder silently passes.

The correct text encoder for Qwen-Image is `Qwen2.5-VL-7B-Instruct-Q4_K_M.gguf` (4.4 GB, hidden dim 3584), which matches the model's architecture. This file is a Qwen2-VL model, not a Qwen3 model — they look similar on HuggingFace but have different hidden dimensions and incompatible embeddings.

## The Working 8 GB Node Chain

```
UnetLoaderGGUF → Qwen_Image_Distill-Q3_K_S.gguf
CLIPLoaderGGUF → Qwen2.5-VL-7B-Instruct-Q4_K_M.gguf (type=qwen_image)
VAELoader      → qwen_image_vae.safetensors
CLIPTextEncode → Chinese prompt (positive)
CLIPTextEncode → "ugly, blurry, text" (negative)
EmptyLatentImage → 1024×1024
KSampler       → steps=20-30, cfg=4.0-5.0, scheduler=sgm_uniform
VAEDecode      → SaveImage
```

Required custom nodes: `ComfyUI-GGUF` (for `UnetLoaderGGUF` and `CLIPLoaderGGUF`).

Optimal parameters for Q3_K_S:

| Parameter | Value | Note |
|-----------|-------|------|
| resolution | 1024×1024 | 768×768 is faster (~150s at 15 steps) |
| steps | 20-30 | 50 steps does not fix blurriness |
| cfg | 4.0-5.0 | Higher CFG doesn't help |
| scheduler | sgm_uniform | Also works: normal |
| speed | ~10s/step @ 768×768 | ~12s/step @ 1024×1024 |

## The Honest Quality Gap

The full model at 9/10 and the GGUF at 6-7/10 is not a small gap. The blurriness is not a parameter problem — it is a quantization ceiling. Q3_K_S is a 3-bit quantization of a distilled model. Pushing to 50 steps, changing the scheduler, or increasing CFG does not recover the detail.

| Metric | Full (12 GB+) | GGUF Q3_K_S (8 GB) |
|--------|:------------:|:-------------------:|
| Chinese text rendering | ✅ Excellent | ⚠️ Usable, edges soft |
| Detail clarity | 9/10 | 6-7/10 |
| Color accuracy | 9/10 | 7/10 |
| Inference time | ~240s @ 1024 | ~240s @ 1024 |
| VRAM | 12 GB+ | 5.4 GB |

The GGUF route's value proposition is not "almost as good as full." It's "runs on 8 GB when nothing else does, and Chinese text is still readable."

## Lessons

| Lesson | Detail |
|--------|--------|
| **Qwen-Image needs Qwen2-VL, not Qwen3** | Hidden dimension mismatch: 3584 ≠ 2560. Causes silent blurriness. |
| **3-bit GGUF is blurry by design** | Q3_K_S caps at 6-7/10 quality. More steps don't fix it. |
| **CFG=5 + normal scheduler** is the full-model sweet spot | Higher CFG and karras make both routes worse. |
| **Qwen VAE is misclassified as WanVAE** | ComfyUI's key-name matching labels it wrong. It works but isn't optimal. |
| **Download >9 GB files via modelscope, not HF** | Xet CDN drops connections. `modelscope download` works where `huggingface-cli` fails. |
| **The full model is 56 GB, not 17 GB** | Many articles cite 17B parameters but forget that bf16 weights are 2× the parameter count. |
