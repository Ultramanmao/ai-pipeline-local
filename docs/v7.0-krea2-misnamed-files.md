# Krea 2 on 8 GB: The Misnamed File That Cost Me a Day

> Date: 2026-07-17 · GPU: RTX 5060 Laptop 8GB · CUDA: cu132 · Framework: ComfyUI · OS: Windows 11 native
> Release: https://github.com/Ultramanmao/ai-pipeline-local/releases/tag/v7.0

## TL;DR

Krea 2 (Krea2Turbo) is one of the fastest open-source text-to-image models available — a compact architecture that competes with Z-Image Turbo on speed while producing clean, detailed images at 1024×1024.

The trap is not the model. It's the files.

Over two days, I discovered:
1. **Two files labeled as Krea 2 VAE and text encoder were actually Wan 2.1 UNETs**, silently misnamed on HuggingFace.
2. **The official GGUF was truncated at 3.84 GB** instead of the expected ~5.2 GB — a download corruption that silently produces garbage tensor metadata.
3. **A ComfyUI API syntax quirk** (`"3:0"` vs `["3", 0]`) breaks dependency chains so completely the pipeline appears to "run" in 0.00 seconds with empty output.

This article documents the full debugging chain, the binary-level diagnosis that identified the misnamed files, the working (but still blocked) node configuration, and what the Krea 2 pipeline actually requires to run on 8 GB.

---

## What Is Krea 2

Krea 2 (also known as Krea2Turbo) is a text-to-image diffusion model from the Krea team, positioned as a lightweight alternative to larger 17B-class models. It uses a Flux-style latent space (16-channel latents, decoder compatible with Flux VAEs) but with a more compact transformer body.

Key architectural details:
| Component | Specification |
|-----------|--------------|
| Architecture | Diffusion transformer (Flux-style) |
| Latent space | 16 channels (compatible with standard Flux VAE) |
| Native resolution | 1024×1024 |
| Text encoder | **Qwen3-VL-4B-Instruct** (single TE, not CLIP + T5) |
| Distinctive feature | Text fusion projector (`txtfusion.projector`) with 12-layer tap output |
| Quantized size | ~5.2 GB (Q3_K_S GGUF) |

The model is particularly interesting for 8 GB GPUs because the compact UNet plus a quantized GGUF should fit comfortably within VRAM limits — if you can get past the file trap.

## Trap 1: The VAE That Wasn't a VAE

On my first attempt, the pipeline loaded every node except the final VAE decode step. Everything appeared correct:

```
UnetLoaderGGUF → krea2_turbo-Q3_K_S.gguf      ✅
CLIPLoaderGGUF → qwen3vl_4b-iq4_xs.gguf       ✅ (apparently)
VAELoaderGGUF  → pig_wan_vae_fp32-f16.gguf    ✅ (apparently)
VAEDecode      → SaveImage                    💥
```

The error from `SaveImage` was:
```
'str' object has no attribute 'shape'
```

This is a signature error pattern — the VAE decode node returned a string error message instead of a tensor, and the downstream `SaveImage` node tried to call `.shape` on it.

**Root cause:** `pig_wan_vae_fp32-f16.gguf` is not a VAE. Despite the filename suggesting a Wan 2.x VAE, I confirmed by reading the GGUF binary header:

```python
import struct
with open('pig_wan_vae_fp32-f16.gguf', 'rb') as f:
    magic = f.read(4)
    # general.architecture identifier
    arch_value = struct.unpack('<I', f.read(4))
```

Result: `general.architecture = 0x1c676970` → **"pig"** — this is the Wan 2.1 diffusion UNET identifier. The file is a full transformer model with diffusion weights, not a VAE decoder.

The real VAE for Krea 2 is the standard **Flux `ae.safetensors`** (320 MB, 16-channel decoder). Krea 2 uses a Flux-compatible 16-channel latent space, so it inherits the Flux VAE. This was easy to get right once I knew it.

## Trap 2: The Text Encoder That Was Also a UNET

With the VAE fixed, the pipeline moved to the text encoder step and crashed again:

```
ValueError: This gguf file is incompatible with llama.cpp!
qwen3vl_4b-iq4_xs.gguf
```

The `gguf_clip_loader` explicitly rejects any GGUF file whose architecture is not CLIP-compatible. The filename claimed this was a Qwen3-VL 4B text encoder in 4-bit integer quantization.

**Root cause: the same binary header diagnostic.**

Reading the GGUF header of `qwen3vl_4b-iq4_xs.gguf`:
```
general.architecture = 0x1c676970 → "pig"
tensor_count = ~713 tensors
```

**This was also a Wan 2.1 UNET.** The file was misnamed entirely — it contained a full diffusion transformer body, not a text encoder.

Two files, both misnamed. Both claimed to be Krea 2 components, both actually Wan 2.1 diffusion models. The mislabeling happened upstream on HuggingFace, where files are occasionally renamed or symlinked incorrectly during repo maintenance.

---

## Trap 3: The Truncated GGUF

After replacing both files, I downloaded the official `krea2_turbo-Q3_K_S.gguf` from the HF mirror. The download completed successfully, but the pipeline crashed at a new stage:

```
TypeError: buffer is too small for requested array
  at shared/qtypes/gguf.py:436 _gguf_open_tensor_numpy()
```

The GGUF parser was trying to read a tensor at an offset that exceeded the file size — meaning the tensor metadata was referencing locations that didn't exist.

**Root cause: download truncation.**

I parsed the GGUF header directly:

| Field | Expected | Actual | Verdict |
|-------|----------|--------|---------|
| File size | ~5.2 GB (Q3_K_S) | 3.84 GB | **Truncated** |
| tensor_count | ~1500-2500 | 1,846,835,937,283 | Garbage |
| meta_count | ~10-50 | 17,179,869,184 | Garbage |
| magic bytes | "GGUF" | "GGUF" | Correct (only first 4 bytes survived) |

The first 4 bytes (the magic header) were correct because they're at the very start of the file. Everything after that — including the tensor offset table and metadata — was garbage data from an incomplete write. The downloader reported success but only wrote 74% of the expected file.

All three copies I had (`unet_gguf/`, `diffusion_models/`, `ckpts/`) traced back to this same truncated source file, so the entire pipeline was built on a corrupted UNet.

## Trap 4: The ComfyUI API Dependency Chain

While debugging the text encoder issue, I noticed something strange: some prompt submissions executed in **0.00 seconds** and returned `executed: []` (empty).

This is the `--cache-none` mode symptom. When ComfyUI can't resolve a node dependency, it silently skips the node and the upstream chain collapses. The root cause was a syntax error in how I passed links in the JSON API:

```json
// WRONG — string format:
"vae": "3:0"

// CORRECT — array format:
"vae": ["3", 0]
```

The `is_link()` function in `graph.py` only recognizes array-formatted links. A string like `"3:0"` is treated as a static value, not a dependency link. The downstream node doesn't pull in its inputs, the chain never expands, and ComfyUI reports an empty execution — making it look like the pipeline ran instantly when it actually did nothing.

This is a ComfyUI JSON API quirk that catches everyone who writes prompts programmatically for the first time.

## The Working Configuration (Blocked)

The correct node configuration for Krea 2 on 8 GB is:

```
UnetLoaderGGUF    → krea2_turbo-Q3_K_S.gguf (Q3_K_S, ~5.2 GB)
CLIPLoaderGGUF    → Qwen3-VL-4B-Instruct TE (safetensors, ~8.9 GB)
    type = "krea2"        ← Required for Krea2 text fusion routing
VAELoader         → ae.safetensors (Flux 16ch VAE, 320 MB)
VAEDecode         → SaveImage
```

The text encoder is the hard requirement: **Qwen3-VL-4B-Instruct** with full tokenizer config (tokenizer.json, tokenizer_config.json, chat_template.jinja). This is not a CLIP model and not compatible with llama.cpp GGUF loading — it must run as safetensors through the native diffusers pipeline.

At 8.9 GB for the TE plus ~2.5 GB for the quantized UNet plus VAE and overhead, the memory budget is tight but feasible on 8 GB if the TE is offloaded to system RAM (which mmgp supports).

**Current status: blocked on the text encoder.** The Qwen3-VL-4B-Instruct model requires HF access permissions that weren't available during this testing window. The UNet and VAE are confirmed correct; the pipeline will work once the proper TE is in place.

## Lessons

| Lesson | Detail |
|--------|--------|
| **Trust the binary header, not the filename** | GGUF `general.architecture` reveals the true model type. "pig" = Wan 2.1 UNET, never a VAE or text encoder. |
| **Verify file size against expected** | A 3.84 GB file claimed to be a 5.2 GB quantized model is truncated. Always check file size after download. |
| **ComfyUI API links must be arrays** | `"3:0"` is silently ignored; `["3", 0]` is the correct format. Empty `executed: []` output usually means a broken dependency chain, not a fast execution. |
| **Krea 2 VAE = Flux VAE** | Krea 2 uses a standard 16-channel Flux-compatible latent space. `ae.safetensors` works. Custom Krea 2 VAEs do not exist. |
| **Qwen3-VL TE is a safetensors-only path** | llama.cpp cannot load it. Requires full tokenizer config and runs through diffusers, not ComfyUI's GGUF loader. |
| **8 GB is feasible with offload** | The UNet fits on GPU; the TE must be CPU-offloaded. mmgp's runtime quantization is designed for exactly this. |

---

## The Real Story

Krea 2 is a genuinely fast model on 8 GB — but the ecosystem around it is thin and the file naming on HuggingFace is unreliable. Two days of debugging came down to reading 4 bytes of binary data.

The takeaway: **never assume a file is what its filename says it is.** Read the header. Check the file size. Verify the architecture. The model will forgive almost anything, but a misnamed file will not.
