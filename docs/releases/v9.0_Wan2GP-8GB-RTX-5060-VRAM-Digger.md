# Wan2GP on 8 GB: The VRAM Digger That Saved My 5060 — One Trap

> Date: 2026-07-18 · GPU: RTX 5060 Laptop 8GB · CUDA: cu132 · Framework: Wan2GP v12.3 (DeepBeepMeep) · OS: Windows 11 native

## TL;DR

Wan2GP is a super-app for the "GPU Poor" — a single web interface that unifies dozens of open-source image, video, audio, and TTS models under one roof, with architecture-aware downloads and deep VRAM optimization.

It is not just another ComfyUI wrapper. It is a **different architecture entirely**: each model is loaded and optimized independently by a custom model handler, rather than being dragged through a generic node graph.

On an 8 GB RTX 5060, the headline is mixed:

- **Image generation: works.** Flux.1/2, Z-Image, Qwen-Image, Krea 2, Ideogram 4 all run natively inside Wan2GP with less VRAM than ComfyUI because the browser can be forced to CPU-only (saving 1–5 GB).
- **Short video generation: mostly works.** LTX-2.3, Wan 2.2 (downsampled), Bernini 1.3B.
- **Long-form video: blocked on 8 GB.** Bernini 14B (14 GB), Bernini v2v (12 GB), Wan 2.2 at native 720p require more VRAM than this card has.
- **The one trap that matters:** Wan2GP requires **Python 3.11** and **PyTorch 2.10.0+cu130**, not the Python 3.13 / cu132 stack the rest of this blog uses. Trying to run Wan2GP in the comfyui venv fails immediately.

## What Is Wan2GP

Wan2GP (by DeepBeepMeep) is a free, open-source super application that consolidates 40+ generative AI models into a single web interface. Unlike ComfyUI, which requires you to wire models together via a node graph, Wan2GP provides a curated, model-specific UI for each generator — text-to-image, image-to-video, video-to-video, inpainting, TTS, audio generation — with per-model prompt enhancers, LoRA management, and reusable templates.

Key architectural choices:

| Aspect | Wan2GP approach | ComfyUI approach |
|--------|----------------|------------------|
| Model loading | Per-model handler (`wan_handler.py`, `ovi_handler.py`, `scail2/`) — each model gets a dedicated loader | Generic `DiffusersLoader` / `UNETLoader` nodes |
| VRAM optimization | Architecture-aware auto-download + Sage/Flash/GGUF kernels baked in | Manual `--lowvram` / `--fp8_e4m3fn-unet` flags |
| UI | Curated tabs per modality (image / video / audio) | Canvas with dragged nodes |
| Prompt handling | Per-model `Prompt Enhancer` + JSON format (Ideogram 4) | Raw string input |
| Browser VRAM | Explicit Chrome no-GPU scripts to reclaim 1–5 GB | No equivalent |
| Updates | One-command `update.bat` / `run.bat` | Manual `git pull` + pip |

The project's official positioning is blunt: **"The best Open Source Generative Models Accessible to the GPU Poor."** The July 2026 release (v12.3, "The VRAM Digger") made browser-side GPU reclamation first-class — a critical feature for 8 GB cards.

## Trap: Wrong Python Version Kills It Immediately

If you try to run `python wgp.py` inside the comfyui venv (Python 3.13, PyTorch 2.13.0+cu132), you hit two failures before any UI appears:

**Failure 1: PyTorch version mismatch.**
Wan2GP's custom kernels (SageAttention, FlashAttention patches, Triton compilation) are compiled against PyTorch 2.10.0+cu130. PyTorch 2.13.0 has a different CUDA runtime ABI. SageAttention fails to compile on cu132 with an internal compiler error that looks like a kernel bug but is actually a version incompatibility.

**Failure 2: `importlib.metadata` changes in Python 3.13.**
Wan2GP's plugin system (`plugins.json`, `PluginLoader`) uses `importlib.metadata.packages_distributions()` in its discovery code. This API was deprecated in Python 3.12 and removed in 3.13, causing a `AttributeError` on startup.

**Solution:** A separate conda environment with Python 3.11:
```bash
conda create -n wan2gp python=3.11
conda activate wan2gp
pip install torch==2.10.0 torchvision==0.25.0 torchaudio==2.10.0 --index-url https://download.pytorch.org/whl/cu130
pip install -r requirements.txt
python wgp.py
```

This is the single most common failure reported in Wan2GP's Discord and the one that costs first-time users the most time. The README's quick-start section says `conda create -n wan2gp python=3.11.14` — the version pin is there, but easy to miss when you're coming from a ComfyUI habit.

## The VRAM Digger: Browser GPU Reclamation

Wan2GP's headline feature for 8 GB GPUs (v12.3, "The VRAM Digger") is an explicit mechanism to disable GPU acceleration in the web browser that serves the Wan2GP UI:

> Run `scripts/start-chrome-no-gpu.bat` (Windows) or `scripts/start-chrome-no-gpu.sh` (Linux) to launch Chrome without using your GPU.

**Why this matters on 8 GB:**

The browser renders the Wan2GP web UI — galleries, model selectors, progress bars, image previews. By default, Chrome uses GPU-accelerated rendering via ANGLE/DirectX, which reserves VRAM as GPU memory. On a 5060, this can be 1–5 GB of VRAM that is simply not available for model inference.

| Browser GPU | VRAM available to Wan2GP |
|-------------|-------------------------|
| Enabled (default) | ~5–6 GB |
| Disabled (no-gpu scripts) | **~7–8 GB** |

This is Wan2GP's most distinctive optimization. No equivalent exists in ComfyUI — ComfyUI's web UI (Gradio) also uses GPU rendering, and there is no built-in script to disable it. The Wan2GP scripts use Chrome flags (`--disable-gpu`, `--disable-gpu-compositing`) to force software rendering for the UI while keeping the underlying model inference on the GPU.

On an 8 GB card, the difference between "model fails to load" and "model loads and runs" is often this 1–5 GB swing.

| Model | Modality | Status | VRAM | Notes |

The pattern is clear: **image generation is fully viable on 8 GB. Short-form video (24–40 frames) is feasible with distilled/small models. Long-form and high-resolution video remain out of reach.**

## Working Configuration

```bash
# 1. Activate the conda environment
conda activate wan2gp

# 2. Disable browser GPU (critical on 8 GB)
start-wan2gp-no-gpu.bat   # or manual: scripts/start-chrome-no-gpu.bat

# 3. Launch Wan2GP
python wgp.py

# 4. In the UI: select model → use Prompt Enhancer → generate
#    Recommended per-model settings for 8 GB:
#    - Flux: GGUF Q4_K_S, LoRA realism, 30 steps
#    - Z-Image: 4 steps, 1312²
#    - LTX-2.3: distilled, 24 frames, downsampled
#    - Bernini: 1.3B variant only
```

Model files are shared with the ComfyUI installation under `E:/WSL/ComfyUI/models/`. Wan2GP's architecture-aware downloader recognizes files already present and skips re-downloading, which saves bandwidth.

## Lessons

| Lesson | Detail |
|--------|--------|
| **Wan2GP is a different Python world** | Requires Python 3.11 + PyTorch 2.10.0+cu130. Cannot coexist with the comfyui venv (Python 3.13 / cu132). A separate conda env is mandatory. |
| **Browser VRAM is real VRAM** | Chrome/Gradio GPU rendering consumes 1–5 GB. Wan2GP's no-gpu scripts are the single most impactful optimization for 8 GB. |
| **Image generation is solved on 8 GB** | All major open-source image models run in Wan2GP with VRAM under 8 GB. The remaining challenge is video. |
| **Long-form video needs a bigger card** | Bernini 14B (14 GB), Wan 2.2 native 720p (10+ GB) are not 8 GB problems. The 8 GB budget for video is distilled 24–40 frame generation. |
| **Wan2GP vs ComfyUI is not replacement** | They coexist: ComfyUI for node-graph experimentation, Wan2GP for curated, low-VRAM, one-click generation. |
| **One-click scripts are real** | `run.bat` / `update.bat` / `manage.bat` handle environment management, updates, and model downloads. No manual `git pull` needed. |

## The Real Story

Wan2GP was built for a specific problem: you want to run a generative AI model, but you don't have a 4090 and you don't want to spend six hours wiring a ComfyUI node graph just to generate one image.

The project's founder, DeepBeepMeep, has built a family of "GP" applications — HunyuanVideoGP, FluxFillGP, Cosmos1GP — each targeting a different model family. Wan2GP is the umbrella that puts them all under one roof.

What makes it genuinely useful for the 8 GB conversation is not the model list (ComfyUI can load most of the same models) — it is the **VRAM engineering**: browser GPU reclamation, per-model optimized loaders, and architecture-aware downloads that know how much VRAM each model actually needs before loading it.

On this 5060, image generation is genuinely solved. Video generation is an ongoing negotiation with VRAM. The question for 2026 is not whether open-source AI fits on consumer hardware — it does, for images. The question is whether distilled models can close the gap for video before the next GPU generation makes 8 GB standard.
